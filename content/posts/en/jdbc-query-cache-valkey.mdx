## Introduction

On March 26, 2026, AWS [announced that the AWS Advanced JDBC Wrapper now supports automatic query caching with Valkey](https://aws.amazon.com/about-aws/whats-new/2026/03/aws-jdbc-caching-with-valkey/). The feature lets you cache JDBC query results in ElastiCache with minimal code changes — just configure the driver and add SQL comment hints like `/* CACHE_PARAM(ttl=300s) */` to queries you want cached.

Previously, implementing query caching required manually writing store/retrieve logic for each query. The Remote Query Cache Plugin absorbs this at the JDBC driver level.

This article shares the results of testing the plugin with Aurora PostgreSQL Serverless v2 and ElastiCache for Valkey Serverless. The key finding: **the first connection to ElastiCache Serverless always times out, causing CacheMonitor to enter an unstable state**. See the official docs at [Caching database query results](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/caching-database-query-results.html) and [Remote Query Cache Plugin](https://github.com/aws/aws-advanced-jdbc-wrapper/blob/main/docs/using-the-jdbc-driver/using-plugins/UsingTheRemoteQueryCachePlugin.md).

## Test Application

The plugin requires only three changes from a standard JDBC connection:

1. Change the URL prefix to `jdbc:aws-wrapper:postgresql://`
2. Set `wrapperPlugins` to `remoteQueryCache`
3. Set `cacheEndpointAddrRw` to the ElastiCache endpoint

```java title="Java (cache connection setup)"
Properties props = new Properties();
props.setProperty("user", "postgres");
props.setProperty("password", password);
props.setProperty("wrapperPlugins", "remoteQueryCache");
props.setProperty("cacheEndpointAddrRw", "my-cache.serverless.apne1.cache.amazonaws.com:6379");

Connection conn = DriverManager.getConnection(
    "jdbc:aws-wrapper:postgresql://my-aurora.cluster-xxx.ap-northeast-1.rds.amazonaws.com:5432/postgres",
    props);
```

To cache a query, add a SQL comment hint with the TTL:

```java title="Java (query hint)"
ResultSet rs = stmt.executeQuery(
    "/* CACHE_PARAM(ttl=300s) */ SELECT * FROM products WHERE category = 'laptop' ORDER BY price DESC");
```

Queries without the hint execute directly against the database as usual.

## Test 1: Basic Cache Behavior

Ran the same query 8 times on a single connection to observe cache behavior.

```text title="Output"
Query 1: 3 rows, 10568 ms
Query 2: 3 rows, 7 ms
Query 3: 3 rows, 7 ms
Query 4: 3 rows, 19 ms
Query 5: 3 rows, 100 ms
Query 6: 3 rows, 216 ms
Query 7: 3 rows, 110 ms
Query 8: 3 rows, 90 ms
```

Query 1 took ~10.5 seconds. The logs reveal a `TimeoutException` on the first cache read, causing CacheMonitor to transition from `HEALTHY` to `SUSPECT`:

```text title="Output (logs)"
[HEALTHY→SUSPECT] jdbc-cache-test-xxx.serverless.apne1.cache.amazonaws.com:6379
  READ failed: CONNECTION - TimeoutException: Request timed out
```

Query 1 fell back to the database (cache miss), while writing the result to the cache in the background. Queries 2-3 at 7ms are suspiciously fast — the CacheMonitor was still in SUSPECT state, so the plugin bypassed the cache entirely and queried the database directly. Queries 5-8 at 90-216ms represent actual cache hit latency.

Additionally, the CacheMonitor health checks consistently failed with a pool borrow timeout:

```text title="Output (logs)"
SEVERE: Non-recoverable error (DATA) for jdbc-cache-test-xxx:6379:
  Timeout waiting for idle object, borrowMaxWaitDuration=PT0.1S
```

The internal borrow timeout is hardcoded at 100ms, which is shorter than the TLS handshake time for ElastiCache Serverless.

## Test 2: Performance Comparison

Compared latency over 20 iterations (after 3 warmup rounds) between plain JDBC and the cache plugin.

| Metric | No Cache | Cached |
| --- | --- | --- |
| Average | 2.1 ms | 106.8 ms |
| Median | 2.0 ms | 105.0 ms |
| Min | 2 ms | 105 ms |
| Max | 3 ms | 132 ms |

**The cached path was ~50x slower** — an unexpected result.

This is specific to the test environment: Aurora Serverless v2 and EC2 are in the same VPC, and the test data is only 10 rows, so direct DB queries complete in 2ms. The TLS overhead to ElastiCache Serverless adds ~100ms per cache operation.

Query caching delivers value when:

- **Queries are expensive** (aggregations, JOINs, non-indexed lookups taking hundreds of ms)
- **Database CPU pressure needs relief** (high-concurrency read offloading)
- **Network latency between app and DB is significant** (cross-AZ or cross-region)

For lightweight queries in the same VPC, the cache overhead dominates.

## Test 3: Cache Staleness and TTL

Updated the database after caching a query to check whether stale data would be returned.

```text title="Output"
Step 1: Prime cache
  Query 1: MacBook Pro 14 = ¥248000.00  (cache MISS → fetched from DB, written to cache)
  Query 2: MacBook Pro 14 = ¥248000.00  (cache HIT)
Step 2: Update price in DB (248000 -> 228000)
Step 3: Verify DB has new value
  DB value: ¥228000.00
Step 4: Read from cache (expect stale 248000)
  Cached value: ¥228000.00  ← Fresh value, not stale
```

**The cache did not return stale data.** The logs show that at Step 4, the CacheMonitor health check error (`Timeout waiting for idle object`) caused the cache read to be skipped, falling back to the database.

When CacheMonitor is unstable, the plugin automatically bypasses the cache and returns results directly from the database. This is correct fail-safe behavior, but it means **you can't tell from the application side whether the cache is actually being used** — only latency changes reveal it.

## Summary

- **The first connection always times out** — The TLS handshake to ElastiCache Serverless exceeds the plugin's internal connection timeout (default 2s). Even increasing `cacheConnectionTimeoutMs` to 10s or 30s doesn't help because the Valkey Glide client has its own timeout. The first query always falls back to the database as a cache miss, adding significant latency at application startup
- **CacheMonitor health checks consistently fail** — The internal connection pool borrow timeout is fixed at 100ms, shorter than ElastiCache Serverless TLS connection establishment. Health checks always report `Non-recoverable error (DATA)`, but the data-path cache reads/writes use a separate path and still function. However, the unstable health state causes intermittent cache bypasses
- **Lightweight queries are slower with caching** — For queries completing in 2ms within the same VPC, cache hits add ~105ms of TLS overhead. Query caching is designed for expensive queries (hundreds of ms or more) or high-concurrency DB offloading scenarios
- **Fail-safe behavior works correctly** — When the cache is unhealthy, the plugin automatically falls back to the database. With `failWhenCacheDown` set to `false` (default), cache issues never cascade into application failures

With ElastiCache Serverless, the initial connection timeout and CacheMonitor instability are significant concerns. A node-based ElastiCache cluster (without TLS, or with pre-warmed connections) may mitigate these issues. I strongly recommend testing with your actual query patterns and latency requirements before production adoption.
