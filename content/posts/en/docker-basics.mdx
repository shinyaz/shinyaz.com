## What is Docker?

Docker is a platform that packages application runtime environments using container-based virtualization. Unlike traditional virtual machines (VMs), containers share the host OS kernel, making them faster to start and lighter on resources.

**Containers vs VMs:**

- **VM**: Runs a full guest OS on top of a hypervisor. Higher overhead
- **Container**: Shares the host OS kernel, isolated at the process level. Lightweight and fast

## Core Concepts

### Images

An image is the blueprint for a container. It contains a filesystem snapshot and execution commands, composed of read-only layers.

### Containers

A container is a running instance created from an image. A writable layer is added on top of the image layers.

### Dockerfile

A text file that describes the steps to build an image.

```dockerfile title="Dockerfile"
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```

What each instruction does:

- `FROM` — Specify the base image
- `WORKDIR` — Set the working directory
- `COPY` — Copy files into the image
- `RUN` — Execute a command during build
- `EXPOSE` — Declare a port as documentation
- `CMD` — Default command when the container starts

## Commonly Used Commands

### Working with Images

```bash
# Build an image
docker build -t myapp .

# Build with a specific tag
docker build -t myapp:v1.0 .

# List images
docker images

# Remove an image
docker rmi myapp

# Pull an image from Docker Hub
docker pull nginx:latest
```

### Working with Containers

```bash
# Run a container (foreground)
docker run -p 3000:3000 myapp

# Run in the background (detached mode)
docker run -d -p 3000:3000 --name my-container myapp

# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Stop a container
docker stop my-container

# Remove a container
docker rm my-container

# View container logs
docker logs -f my-container

# Open a shell inside a running container
docker exec -it my-container sh
```

### Volumes and Networks

```bash
# Create a named volume
docker volume create mydata

# Run with a volume mounted
docker run -v mydata:/app/data myapp

# Bind mount a host directory
docker run -v $(pwd)/src:/app/src myapp

# Create a custom network
docker network create mynet

# Run on a specific network
docker run --network mynet --name api myapp
```

## Docker Compose

A tool for defining and running multi-container applications. Configuration is written in `compose.yml`.

```yaml title="compose.yml"
services:
  app:
    build: .
    ports:
      - "3000:3000"
    volumes:
      - ./src:/app/src
    depends_on:
      - db
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb

  db:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb

volumes:
  pgdata:
```

```bash
# Build and start all services
docker compose up -d

# View logs
docker compose logs -f

# Stop and remove containers
docker compose down

# Remove volumes as well
docker compose down -v
```

## Multi-stage Builds

Separate the build environment from the runtime environment to reduce the final image size.

```dockerfile title="Dockerfile"
# Build stage
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Runtime stage
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/package*.json ./
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/public ./public
EXPOSE 3000
CMD ["npm", "start"]
```

This approach keeps build-only tools and source code out of the final image.

## Best Practices

- **Use `.dockerignore`** — Exclude `node_modules`, `.git`, `.next`, etc. from the build context to speed up builds
- **Leverage layer caching** — Copy things that change less often (dependency installation) first, and things that change frequently (source code) last
- **Use lightweight base images** — Choose `alpine` variants to significantly reduce image size
- **One process per container** — Each container should have a single responsibility. Use Docker Compose to separate multiple processes
- **Don't run as root** — Use the `USER` instruction to switch to a non-privileged user

## Summary

Docker lets you develop, test, and deploy applications in a consistent environment. The standard workflow is to define build steps in a Dockerfile and manage multi-service setups with Docker Compose. As a next step, consider learning container orchestration with Kubernetes.
