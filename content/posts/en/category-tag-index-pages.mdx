## The Problem

This blog already had individual category pages (`/en/category/programming`) and tag pages (`/en/tag/nextjs`). However, there was **no index page to browse all categories or tags** at a glance. Users could navigate to specific categories or tags via badges on individual posts, but there was no way to explore the blog's overall content structure.

This update adds four new pages:

- `/en/category` — English category index
- `/ja/category` — Japanese category index
- `/en/tag` — English tag index
- `/ja/tag` — Japanese tag index

## Design Approach

This blog uses Velite to statically generate data from MDX files at build time, with helper functions in `src/lib/posts.ts` to query the data. All the functions needed for index pages already existed:

- `getAllCategories()` — fetch all categories
- `getCategoryName()` / `getCategoryDescription()` — bilingual category name and description
- `getPostsByCategory()` — fetch posts by category (used for post counts)
- `getAllTags(locale)` — fetch all tags by locale
- `getPostsByTag()` — fetch posts by tag (used for post counts)

No new components or utilities were needed — just composing existing functions into new pages.

## Category Index Page

The category index displays a **card-based grid layout**. Each card shows the category name, description, and post count, and links to the individual category page.

```tsx title="src/app/[locale]/category/page.tsx"
import Link from "next/link";
import { getAllCategories, getPostsByCategory, getCategoryName, getCategoryDescription } from "@/lib/posts";
import { locales, isValidLocale, getDictionary } from "@/lib/i18n";

export async function generateStaticParams() {
  return locales.map((locale) => ({ locale }));
}

export default async function CategoryIndexPage({ params }: CategoryIndexPageProps) {
  const { locale } = await params;
  if (!isValidLocale(locale)) notFound();
  const t = getDictionary(locale);
  const categories = getAllCategories();

  return (
    <div className="mx-auto max-w-3xl px-4 py-6 md:py-12">
      <header className="mb-8">
        <h1 className="text-2xl font-bold tracking-tight md:text-3xl">
          {t.category.title}
        </h1>
        <p className="mt-2 text-muted-foreground">{t.category.description}</p>
      </header>
      <div className="grid gap-4 sm:grid-cols-2">
        {categories.map((category) => {
          const name = getCategoryName(category, locale);
          const description = getCategoryDescription(category, locale);
          const postCount = getPostsByCategory(category.slug, locale).length;
          return (
            <Link
              key={category.slug}
              href={`/${locale}/category/${category.slug}`}
              className="rounded-lg border border-border p-4 transition-colors hover:bg-accent"
            >
              <h2 className="font-semibold">{name}</h2>
              {description && (
                <p className="mt-1 text-sm text-muted-foreground">{description}</p>
              )}
              <p className="mt-2 text-xs text-muted-foreground">
                {t.category.postCount.replace("{count}", String(postCount))}
              </p>
            </Link>
          );
        })}
      </div>
    </div>
  );
}
```

Key points:

- Implemented as a **server component** — no `"use client"` needed since there's no interactivity
- `generateStaticParams` generates pages for each locale at build time
- `sm:grid-cols-2` creates a responsive grid: single column on mobile, two columns on `sm` and above
- `getCategoryName()` / `getCategoryDescription()` automatically switch between languages

## Tag Index Page

The tag index uses a **tag cloud style flex layout**. Each tag displays its name with a post count in parentheses.

```tsx title="src/app/[locale]/tag/page.tsx"
export default async function TagIndexPage({ params }: TagIndexPageProps) {
  const { locale } = await params;
  if (!isValidLocale(locale)) notFound();
  const t = getDictionary(locale);
  const tags = getAllTags(locale);

  return (
    <div className="mx-auto max-w-3xl px-4 py-6 md:py-12">
      <header className="mb-8">
        <h1 className="text-2xl font-bold tracking-tight md:text-3xl">
          {t.tag.indexTitle}
        </h1>
        <p className="mt-2 text-muted-foreground">{t.tag.indexDescription}</p>
      </header>
      <div className="flex flex-wrap gap-3">
        {tags.map((tag) => {
          const postCount = getPostsByTag(tag, locale).length;
          return (
            <Link
              key={tag}
              href={`/${locale}/tag/${tag}`}
              className="inline-flex items-center gap-1.5 rounded-full border border-border px-3 py-1.5 text-sm text-muted-foreground transition-colors hover:bg-accent hover:text-accent-foreground"
            >
              #{tag}
              <span className="text-xs">({postCount})</span>
            </Link>
          );
        })}
      </div>
    </div>
  );
}
```

The styling is slightly larger than the existing `TagBadge` component (`px-2.5 py-0.5 text-xs`) — using `px-3 py-1.5 text-sm` for better readability and tap targets on the index page.

A key design difference from the category index: `getAllTags(locale)` takes a locale argument. Tags are dynamically collected from post frontmatter, so Japanese and English posts may have different tag sets. Categories, on the other hand, are defined in static YAML files and don't vary by locale.

## SEO

Both pages follow the established patterns for SEO:

- `generateMetadata` sets `title`, `description`, `openGraph`, and `alternates`
- `buildAlternateLanguages` generates `hreflang` alternate language links
- Explicit `canonical` URLs

```typescript
export async function generateMetadata({ params }: CategoryIndexPageProps): Promise<Metadata> {
  const { locale } = await params;
  if (!isValidLocale(locale)) return {};
  const t = getDictionary(locale);
  return {
    title: t.category.title,
    description: t.category.description,
    openGraph: {
      title: t.category.title,
      description: t.category.description,
      url: `${SITE_URL}/${locale}/category`,
      type: "website",
    },
    alternates: {
      canonical: `${SITE_URL}/${locale}/category`,
      languages: buildAlternateLanguages((l) => `/${l}/category`),
    },
  };
}
```

```typescript title="__tests__/lib/i18n.test.ts"
  expect(en.category.title).toBe("Categories");
  expect(en.category.postCount).toContain("{count}");

it("has tag index keys", () => {
  const ja = getDictionary("ja");
  const en = getDictionary("en");
  expect(ja.tag.indexTitle).toBe("タグ");
  expect(en.tag.indexTitle).toBe("Tags");
});
```

### E2E Tests

Playwright tests verify page rendering, navigation, and post counts for both locales.

```typescript title="e2e/category-index.spec.ts"
test.describe("Category Index Page", () => {
  test("English category index page loads with categories", async ({ page }) => {
    await page.goto("/en/category");
    await expect(page.locator("h1")).toContainText("Categories");
    await expect(page.locator("a[href*='/en/category/']").first()).toBeVisible();
  });

  test("each category card shows post count", async ({ page }) => {
    await page.goto("/en/category");
    const firstCard = page.locator("a[href*='/en/category/']").first();
    await expect(firstCard).toContainText(/\d+ post/);
  });

  test("clicking a category navigates to category detail page", async ({ page }) => {
    await page.goto("/en/category");
    await page.locator("a[href*='/en/category/']").first().click();
    await expect(page).toHaveURL(/\/en\/category\/.+/);
  });
});
```

The tag index page has a similar test structure. Both test suites also verify that the blog listing page links correctly to the index pages.

## Summary

By reusing existing helper functions and the data pipeline, category and tag index pages were added with minimal code changes:

- **Category index**: Card-based grid layout with name, description, and post count
- **Tag index**: Tag cloud style flex layout with tag name and post count
- **Blog page**: Header links to category and tag indexes
- **Bilingual support**: i18n dictionary extension for both Japanese and English

Both pages are pure server components, so no additional client JavaScript is shipped. They use Velite's statically generated data directly, with no runtime data fetching.
