## Dockerとは

Dockerは、コンテナ型仮想化技術を使ってアプリケーションの実行環境をパッケージングするプラットフォームです。従来の仮想マシン(VM)と異なり、ホストOSのカーネルを共有するため、起動が高速でリソース消費も少ないのが特徴です。

**コンテナとVMの違い:**

- **VM**: ハイパーバイザ上にゲストOSごと載せる。オーバーヘッドが大きい
- **コンテナ**: ホストOSのカーネルを共有し、プロセスレベルで分離する。軽量で高速

## 基本概念

### イメージ

イメージはコンテナの設計図です。ファイルシステムのスナップショットと実行コマンドを含み、読み取り専用のレイヤーで構成されています。

### コンテナ

イメージから起動した実行中のインスタンスです。イメージの上に書き込み可能なレイヤーが追加されます。

### Dockerfile

イメージのビルド手順を記述するテキストファイルです。

```dockerfile title="Dockerfile"
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```

各命令の意味:

- `FROM` — ベースイメージの指定
- `WORKDIR` — 作業ディレクトリの設定
- `COPY` — ファイルをイメージにコピー
- `RUN` — ビルド時にコマンドを実行
- `EXPOSE` — ドキュメントとしてポートを宣言
- `CMD` — コンテナ起動時のデフォルトコマンド

## よく使うコマンド

### イメージの操作

```bash
# イメージをビルド
docker build -t myapp .

# タグを付けてビルド
docker build -t myapp:v1.0 .

# イメージ一覧を表示
docker images

# イメージを削除
docker rmi myapp

# Docker Hubからイメージを取得
docker pull nginx:latest
```

### コンテナの操作

```bash
# コンテナを起動（フォアグラウンド）
docker run -p 3000:3000 myapp

# バックグラウンドで起動（デタッチモード）
docker run -d -p 3000:3000 --name my-container myapp

# 実行中のコンテナ一覧
docker ps

# 全コンテナ一覧（停止中を含む）
docker ps -a

# コンテナを停止
docker stop my-container

# コンテナを削除
docker rm my-container

# コンテナのログを確認
docker logs -f my-container

# 実行中のコンテナに入る
docker exec -it my-container sh
```

### ボリュームとネットワーク

```bash
# 名前付きボリュームを作成
docker volume create mydata

# ボリュームをマウントして起動
docker run -v mydata:/app/data myapp

# ホストのディレクトリをバインドマウント
docker run -v $(pwd)/src:/app/src myapp

# カスタムネットワークを作成
docker network create mynet

# ネットワークを指定して起動
docker run --network mynet --name api myapp
```

## Docker Compose

複数のコンテナをまとめて管理するためのツールです。`compose.yml` に構成を記述します。

```yaml title="compose.yml"
services:
  app:
    build: .
    ports:
      - "3000:3000"
    volumes:
      - ./src:/app/src
    depends_on:
      - db
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb

  db:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb

volumes:
  pgdata:
```

```bash
# 全サービスをビルド & 起動
docker compose up -d

# ログを表示
docker compose logs -f

# 停止 & コンテナ削除
docker compose down

# ボリュームも含めて削除
docker compose down -v
```

## マルチステージビルド

ビルド環境と実行環境を分離して、最終イメージのサイズを削減できます。

```dockerfile title="Dockerfile"
# ビルドステージ
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 実行ステージ
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/package*.json ./
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/public ./public
EXPOSE 3000
CMD ["npm", "start"]
```

この手法により、ビルド時にしか使わないツールやソースコードが最終イメージに含まれなくなります。

## ベストプラクティス

- **`.dockerignore` を活用する** — `node_modules`, `.git`, `.next` などをビルドコンテキストから除外し、ビルドを高速化する
- **レイヤーキャッシュを意識する** — 変更頻度の低いもの（依存関係のインストール）を先に、変更頻度の高いもの（ソースコード）を後にコピーする
- **軽量なベースイメージを使う** — `alpine` バリアントを選ぶとイメージサイズを大幅に削減できる
- **1コンテナ1プロセスの原則** — コンテナは単一の責務を持つべき。複数プロセスが必要な場合は Docker Compose で分離する
- **root ユーザーで実行しない** — `USER` 命令で非特権ユーザーに切り替える

## まとめ

Dockerはアプリケーションの開発・テスト・デプロイを一貫した環境で行えるツールです。Dockerfile でビルド手順を定義し、Docker Compose で複数サービスを管理するのが基本的なワークフローです。次のステップとして、Kubernetesを使ったコンテナオーケストレーションを学ぶとよいでしょう。
